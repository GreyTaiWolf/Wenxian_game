# AI 与开发协作规则

本项目是一个移动端优先的修仙文字 RPG，当前技术栈为 Vite + React + TypeScript。后续开发需要同时维护工程质量、玩法设计和策划数值，不能只改代码不改文档。

## 上下文与文档恢复规则

- 每次开始实现前，先检查本文件，并按改动类型对照 `docs/GAME_DESIGN.md` 和 `docs/BALANCE.md`。
- 当对话上下文变长、发生压缩或不确定既有规则时，以本文件和 `docs/` 文档为准，不凭记忆继续改。
- 实现完成后，如果改动影响玩法、UI 规则、数值、物品、系统或存档结构，必须在最终回复中说明已同步哪些文档。

## 项目定位

- 游戏核心体验是修炼、突破、背包、历练、大世界探索、回合制战斗、宗门成长、洞府养成和本地存档。
- 默认界面面向手机宽度，主容器约 `430px`，优先保证移动端阅读、点击和信息密度。
- 玩家可见文本使用中文，代码 ID 使用英文 `lower_snake_case`。
- 所有源码和文档保存为 UTF-8，避免中文乱码。

## 目录职责

- `src/types.ts`：共享类型和状态结构。新增系统、模块、解锁项、战斗效果或存档字段时先检查这里。
- `src/data/*.ts`：游戏配置数据，包括境界、技能、物品、敌人、世界、任务和大世界州域。内容和数值优先放这里。
- `src/game/*.ts`：纯游戏逻辑，包括状态变更、存档、战斗引擎和 AI 行动。避免把复杂规则写在组件里。
- `src/components/*.tsx`：界面展示和轻量交互。组件可以调用 `src/game` 的状态变更函数，但不应承担核心数值公式。
- `src/components/GameIcon.tsx`：轻量图标语义层。业务组件使用语义名，不直接散落引用具体图标组件。
- `src/styles.css`：全局视觉样式。保持现代玄幻 App 方向：深色石墨底、半透明面板、细描边、柔和灵光和紧凑 HUD 信息层级。
- `docs/GAME_DESIGN.md`：长期设计文档，记录玩法目的、系统关系、解锁和未来内容。
- `docs/BALANCE.md`：策划数值文档，记录成长、公式、奖励、价格和调参记录。

## 强制文档同步

每次改动若影响玩法、内容、解锁、地图、物品、技能、敌人、奖励、成本、概率或公式，必须同步更新对应文档。

- 影响玩法体验、系统关系、世界内容或未来规划：更新 `docs/GAME_DESIGN.md`。
- 影响数值、公式、成本、掉落、奖励、价格、属性或成功率：更新 `docs/BALANCE.md`。
- 只改 UI 样式、布局、命名整理或纯代码重构时，可以不更新设计和数值文档，但仍需遵守本文件规则。
- 新增系统或模块时，必须同时检查类型定义、数据配置、UI 入口、设计说明和数值说明。
- 新增物品、技能、敌人、任务或地图点时，必须检查 ID 引用、奖励来源、解锁条件和文档记录。

## 内容扩展规则

- 新境界：更新 `src/data/progression.ts`，必要时扩展 `UnlockKey`，并在 `docs/BALANCE.md` 记录修为需求、突破成本和成功率。
- 新系统入口：更新 `PrimaryModule`、`UnlockKey`、底部导航、对应面板和境界解锁配置。
- 新物品：更新 `src/data/items.ts`，确认来源、用途、价格、消耗场景和是否进入商店或任务。
- 新技能：更新 `src/data/skills.ts`，确认可用角色、目标类型、灵力消耗、威力、效果类型、AI 权重和解锁境界。
- 新敌人或战斗组：更新 `src/data/enemies.ts`，确认属性、技能、奖励和目标难度。
- 新地图内容：更新 `src/data/world.ts` 或 `src/data/worldMap.ts`，确认州域、地点、场景、行动类型和奖励。
- 新任务：更新任务配置，确认接取条件、完成条件、奖励、关联地图行为和文档说明。

## 实现风格

- 保持 TypeScript `strict` 下可构建。
- 状态更新保持不可变写法，不直接修改传入的 `game`、`rootSave` 或数组对象。
- 优先复用现有 helper，如 `appendLog`、`addItems`、`removeItems`、`addRewards`、`getRealm`、`getItem`、`getSkill`。
- 避免新增重型状态库或 UI 框架，除非功能复杂度已经明显超过当前结构。
- 不把关键数值硬编码在组件里；组件里的临时展示数值如果变成真实规则，应迁移到 `src/data` 或 `src/game`。
- 存档结构变更必须考虑兼容旧存档，必要时提升版本并写迁移逻辑。

## UI 与体验规则

- 保持移动端优先，按钮最小高度约 `42px`，文本不能溢出或互相遮挡。
- 视觉方向为现代玄幻 App，减少传统水墨、毛笔、卷轴和纸感表达，优先使用清晰的暗色手游 HUD。
- 主题变量集中维护在 `src/styles.css` 的 `:root`，不要在组件或局部样式里零散硬编码主色体系。
- 轻量图标统一使用 `lucide-react` 加 `GameIcon` 语义层；新增图标先补语义名，再在业务组件里引用。
- 修仙专属图标可后续用少量本地 SVG 或图片补足，但必须保持尺寸、描边、颜色和可访问性一致。
- 已锁定功能可以展示入口，但必须明确禁用状态或给出日志反馈。
- 底部导航只放核心模块，不为临时内容增加永久入口。
- 玩家操作应尽量产生日志或场景反馈，保证文字 RPG 的连续叙事感。
- 大世界地图相关交互要保留缩放、拖拽、州域选择和未开放提示。

## 验证要求

- 文档-only 改动无需运行构建。
- 代码、类型、数据或数值改动后，最低验证为 `yarn build`。
- 战斗相关改动需要手动检查开始战斗、使用技能、防御、逃跑、使用物品、胜利奖励和失败回城。
- 存档相关改动需要检查新建存档、继续游戏、自动保存、删除存档和旧数据回退。
- 地图和任务相关改动需要检查地点切换、场景行动、任务接取、完成条件和奖励到账。
